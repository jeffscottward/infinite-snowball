# Infinite Snowball Phase 07 — Secure Offline Installer, Catalog, Store Data

Phase ID: P07
Status: Planned
Owner role: Secure offline catalog/store data engineer
Depends on: P05, P06

## Goal and user value

Build the secure offline content-runtime foundation that lets the game show a cached catalog instantly, install curated immutable assets safely, survive quota/network failures, and preserve playable content offline. This phase starts only after the 90-second vertical slice, input parity, and P05-owned offline PWA baseline pass, so the store reinforces a proven game loop instead of hiding an unfinished one.

## Prerequisites and dependencies

- P05 has passed `VG-05-PLAYABLE`, `VG-05-INPUT-PARITY`, `VG-05-OFFLINE-RUN`, supplemental `VG-05-CSP-WASM`, accessibility, telemetry, and playtester gates, including Dexie v1 persistence, artifact/policy/probe/WASM hash handoff, and the explicit automated-installability-versus-real-install boundary; P07 must not start before this evidence exists.
- P05 hands off the minimal private shell, logical `/play` hash route, web manifest with stable `id: "./"`, current `start_url: "./#/play"`, `scope: "./"`, relative URLs, scope-relative SW registration/precache, Dexie v1, the inert same-origin external CSP probe path/hash, `offline-persistent-profile.spec.ts`, `production-csp-wasm.spec.ts`, and `base-neutral-hosting.spec.ts`. The latter serves each candidate's checksum-identical bytes on isolated root/project-prefix origins and asserts prefix-correct registration/controller identity; there is no P01 worker to consume.
- P06 has passed `VG-06-CLI`, `VG-06-AUTHOR-JOURNEY`, and `VG-06-SUBMISSION`, including successful and rejected fixtures, stable rule IDs, exact package records, packed-file allowlist evidence, and catalog verification output shape.
- P02 protocol/security/offline design remains the source of truth for manifest validation, no-code boundary, install transaction invariants, and error codes.
- P08 will later build visible store UI using the data/controller states produced here; P07 owns data, cache, installer, and controller state, not final presentation.

## Stable inputs and contracts

- Dexie tables: `catalogSnapshots`, distinct `catalogEntries`, distinct `packages`, `assets`, `packageAssets`, `packageLocks`, `installTransactions`, `saves`, `settings`, and `migrations`.
- P05 Dexie v1 migration contract: preserve existing `saves` and `settings` rows atomically before adding catalog/install/export tables; migration failure leaves the P05 v1 database usable.
- Install states: `planned -> staging -> verifying -> committing -> installed`, or `failed`/`canceled` with staging-byte cleanup only, previous lock preservation, and retained transaction rows containing error, verified set, rollback actions, and reconciliation status.
- Cache Storage owns immutable URL responses; Dexie owns metadata, dependency graph, status, active pointers, reference counts, saves, settings, export metadata, and migrations.
- Service-worker strategy: extend the P05 worker's shell/minimal-starter precache with stale-while-revalidate catalog/thumbnails, cache-first immutable assets, network-first mutable metadata with last-valid fallback, version namespaces, and one known-good shell. The snapshot literal `resourceBasePath: "./catalog/"` defines the only v1 resource subtree. Validate each catalog/thumbnail/package/asset reference as relative, resolve it against `self.registration.scope` plus `catalog/`, and fail closed unless it remains same-origin inside that subtree; no root-absolute, deployment-specific, or external CDN/base URL is allowed. After each SW/DB/catalog/package or other build-affecting migration, rerun inherited P05 offline and CSP/inline-WASM/probe regressions on isolated origins.
- SaveExport policy: versioned export/import is local-only, size-bounded, checksum-verified, atomic, and fail-closed for corrupt, oversize, incompatible, rollback, or privacy/non-egress failures; exports exclude credentials, analytics, cloud IDs, undeclared network refs, and local-audio bytes/tags/artwork.
- Security policy: resolve exact acyclic DAG, preflight quota/request persistence, streamed hash/type/size verification, no browser npm/tarball/code execution, no undeclared URLs, per-file validation, and stable P02/P06 rule IDs. Browser public exports and transitive imports must exclude the build-only materializer plus all Node/filesystem modules.
- Offline policy: offline install only if every locked object is already verified/cached; uninstall deletes only reference-count-zero assets; reconciliation repairs drift.
- Catalog artifact contract: the Node-only `tools/catalog-materializer/**` deterministically transforms approved source `catalog/**` into `apps/web/public/catalog/**`; normalized sorting, canonical JSON, hashes, and a clean-output rebuild produce byte-identical files and reject stale/extra output. It is never exported from content-runtime or imported by web/service-worker code. Page and worker runtime resolvers remain browser-safe and map validated records to corresponding same-origin prefix-local files under `catalog/`.

- Web package handoff: after `packages/content-runtime/**` exists, P07 may add `@infinite-snowball/content-runtime` and update only the phase-current allowlist in `apps/web/tests/web-package-contract.test.ts`. It must preserve every permanent private/existing-only/script/boundary assertion, existing P05 scripts/route/offline behavior, and the absence of `@infinite-snowball/ui`.
- Every P07 UI port exposes `getSnapshot(): Readonly<Snapshot>`, `subscribe(listener): () => void`, and `dispose()`. Repeated reads are `Object.is`-identical until a commit; each monotonic `stateVersion` creates one immutable snapshot and exactly one notification. Snapshots include authoritative state, primary `busyCommand|null`, `cancelRequested`, and sanitized stable errors. Async commands return a typed committed version/transaction ID or stable failure (`busy`, `invalid-state`, `offline`, `quota`, `integrity`, `rollback`, `unsupported`, `canceled`, or operation-specific failure).
- Concurrency matrix: duplicate or conflicting start/refresh/install/update/export/import/apply/retry commands while a primary operation is active return `busy` with no side effects. A cancellation targeting that exact active install/update transaction, or `cancelImport` targeting the active staged/import operation, is the only admitted concurrent command; mismatched cancellation returns `invalid-state`. Matching cancellation is idempotent, does not replace the primary `busyCommand`, sets `cancelRequested`, and all duplicate cancel callers resolve to the same terminal outcome only after staging bytes/listeners are cleaned and the retained canceled audit snapshot commits. Retry is admitted only from a matching terminal failed/canceled state with no active primary operation.
- `StoreDataPort` exposes cached/refresh/install/update/cancel/retry states and `refreshCatalog`, `install(packageRef)`, `update(packageRef)`, `cancel(transactionId)`, and `retry(transactionId)`. Cancel during staging, download, or verification follows the concurrency matrix; completion/failure preserves the previous known-good catalog/lock and reports transaction evidence without raw URLs or secrets.
- `SaveDataPort` exposes export/import/confirmation/rollback states and `beginExport`, `stageImport(source)`, `confirmImport`, `cancelImport`, and `retryRollback`. Staging validates before mutation; destructive import cannot pass confirmation implicitly; matching cancel during staging or import cleans temporary data and commits a canceled state before resolving while preserving the prior save/lock snapshot.
- `PwaDataPort` exposes shell/starter health, persistence, known-good fallback, eligibility/prompt/update states and `requestInstall`, `dismissInstall`, `applyUpdate`, `deferUpdate`, and `retryUpdate`. The thin browser adapter captures platform events but never leaks raw events to P08; gesture-required commands resolve only from the initiating user action, and its conflicting starts/retries follow the same busy/terminal-state rules.
- `dispose()` removes listeners/subscriptions and aborts only P07-owned in-flight observation; it never deletes committed data. P08 must wait for typed command completion and authoritative snapshots rather than poll, infer success, or access Dexie/service-worker internals.

## Rollback and recovery notes

All migrations are atomic, idempotent, and reversible to one known-good shell while preserving P05 saves/settings. Failed or canceled installs delete staging bytes only and retain audit rows; bad catalogs keep the last valid snapshot; bad imports fail before mutation without egress. Roll back only P07 package/test/SW/DB/catalog changes while preserving the P05 shell and unified package-test permanent assertions. After rollback, clean-materialize the catalog twice and require byte identity/no stale output, rebuild once, serve checksum-identical bytes on isolated root/project-prefix origins, then rerun the unified package test, `offline-persistent-profile.spec.ts`, `base-neutral-hosting.spec.ts`, prefix-local catalog resolution, service-worker recovery, store offline install, and save portability. Do not hand off to P08 until manifest identity, catalog containment, worker scope, profile persistence, last-valid recovery, and package allowlist evidence all pass.
