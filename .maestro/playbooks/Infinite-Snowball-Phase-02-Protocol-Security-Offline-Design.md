# Infinite Snowball Phase 02 — Protocol, Security, and Offline Transaction Design

Phase ID: P02
Status: Planned
Owner role: Protocol, security, and offline-design engineer
Depends on: P01

## Goal and user value

Define the closed-world content protocol, threat model, no-code package boundary, and offline install transaction design before any browser store or full catalog implementation exists. Users and creators get a safer ecosystem because packages are declarative, hash-verified, provenance-reviewed, recoverable offline, and never arbitrary JavaScript in the browser.

## Prerequisites and dependencies

- P01 workspace, type/test harness, required check names, and lockfile baseline are complete and verified.
- Use the reconciled authoring brief decisions 15–21, settled content/offline/security research, approved ADRs, and system-contract inputs as the controlling security contract for provenance fields, license allowlist, evidence states, budgets, withdrawal, and local audio boundaries.
- Do not treat P03 as a prerequisite: P03 is downstream of P02 and consumes these protocol/security contracts for asset and music implementation after the contract is frozen.
- Use P04/P05 runtime needs only as data consumers; do not implement engine, input, gameplay, or store UI here.
- External account credentials, npm publishing, CDN deployment, signatures/TUF, D1, and cloud services are not prerequisites.

## In scope

- Future `packages/protocol/**` source, tests, fixtures, generated JSON Schema, and browser-safe validators/types.
- Strict manifest definitions for the six content kinds: `level`, `character`, `object-pack`, `campaign`, `music`, `bundle`.
- Stable domain contracts for `PackageRef`, `AssetRecord`, `Provenance`, declarative entries, `CatalogSnapshot`, `CatalogEntry`, `PackageLock`, `InstallPlan`, `InstallTransaction`, `InstallRecord`, and `SaveExport` shapes.
- Threat model and policy docs under future `docs/security/**`, `docs/content-policy/**`, and `docs/offline/**` when generated by the implementation phase.
- Adversarial protocol fixtures under `tests/fixtures/protocol/**` covering traversal, executable content, MIME mismatch, hash mismatch, GLB external references, unsupported codecs, invalid licenses, invalid semver, unknown fields, oversized archives, compression bombs, dependency cycles, and offline-install failures.
- Offline transaction state-machine design for planned -> staging -> verifying -> committing -> installed / failed / canceled, including quota, persistence request, side-by-side update, reference counts, rollback, reconciliation, and last-known-good behavior.
- Explicit v1 no-backend and later-D1 boundary ADR/policy.

## Non-goals

- No full store/catalog runtime implementation, service-worker implementation, Dexie migrations, browser UI, npm resolver, tarball extractor, package publisher, CDN upload, content conversion pipeline, asset acquisition, gameplay, or deployment.
- No executable plugin model, browser npm search, browser tarball handling, browser `import()` of community packages, `eval`, JS/WASM/HTML/CSS community content, signatures/TUF, OPFS dependency, or cloud sync.
- No Supabase/backend/account/cloud save/leaderboard/multiplayer in v1.
- No legal clearance claim; the protocol enforces evidence and fail-closed policy, while human review remains required.

## Stable inputs and contracts

- Browser imports only protocol validators, types, and generated schema artifacts that contain no Node, npm, archive, filesystem, network-following, lifecycle, or code-execution logic.
- Manifest is strict and rejects unknown fields. Required fields include `schemaVersion`, exact `name`, exact `version`, `kind`, engine range, localized metadata, author/homepage/repository/screenshots/icon/tags, SPDX package license, declarative entries, exact dependencies/optional peers, totals, empty v1 capabilities, and flat asset inventory.
- Asset inventory fields include normalized path, MIME, bytes, SHA-256, role, license, provenance, and must support the settled evidence state model handed to P03: `verified`, `incomplete`, `disputed`, `withdrawn`.
- Path validation rejects traversal/absolute/drive/UNC/NUL/encoded variants, symlinks, Unicode/case collisions, unknown files, executables, JS/WASM/HTML/CSS, MIME mismatch, invalid hashes/semver/licenses, unsupported codecs, external/data/network GLB references, and bounded extraction violations.
- V1 uses no catalog CDN/base URL. `CatalogSnapshot.resourceBasePath` is literal `"./catalog/"`. A dedicated `CatalogResourcePathSchema` accepts canonical ASCII relative segments and rejects absolute/root/scheme-relative paths, colon/schemes, backslashes, dot segments, empty segments, percent/encoded variants, query, and fragment aliases.
- Catalog URL-addressable fields are explicit: `CatalogEntry.screenshots`, `CatalogEntry.icon`, and `CatalogEntry.packageRecordPath` use `CatalogResourcePathSchema`; `ImmutableFile.resourcePath` and `CatalogPackageAsset.resourcePath` replace legacy absolute `url`; package-local `path` remains a distinct safe relative path. Generated schemas, fixtures, types, and handoff hashes must contain no legacy `cdnBaseUrl` or catalog asset `url`.
- Cache Storage owns verified responses addressed by resolved prefix-local URLs; Dexie owns metadata/locks/saves/settings/refs/migrations. A browser-safe protocol resolver accepts only validated resource paths and an injected app base; page and worker callers inject bases derived from `document.baseURI` and `self.registration.scope`. Both root and project-prefix cases must remain same-origin inside their own `catalog/` child.
- P07's worker design uses `injectManifest`, shell/starter precache, catalog strategies, version namespaces, health-gated activation, and one known-good shell. Root-absolute, origin-changing, path-escaping, or deployment-specific catalog/SW URLs fail closed. An external CDN is outside v1 and requires a later ADR/schema/CSP/availability migration.

## Test-first acceptance criteria

- `VG-02-SCHEMA`: golden fixtures fail first and pass only when all six kinds and strict records validate, generated JSON Schema matches source, catalog base is literal, URL-addressable fields use the dedicated relative schema, legacy absolute fields are rejected, and unknown fields remain rejected.
- `VG-02-NO-CODE`: browser-boundary tests fail first and pass only when browser exports contain validators/types/schema data plus the pure relative resolver, with no Node/npm/archive/filesystem/lifecycle/eval/community-code path.
- `VG-02-THREATS`: adversarial fixtures fail before policy implementation and pass only when every hostile input produces the correct stable rule ID and fail-closed result.
- `VG-02-OFFLINE-DESIGN`: transaction model tests fail before the state machine is defined and pass only when every failure point preserves the previous lock, deletes staging, avoids orphan refs, and reports actionable recovery state.
- `VG-02-D1-BOUNDARY`: ADR/policy tests fail before no-backend-v1 is documented and pass only when D1/backend work is blocked absent the specified post-v1 evidence.

## Completion and stop condition

P02 is complete only when every gate passes, generated schemas/handoff hashes are deterministic, browser exports are code-execution/filesystem-free, and downstream contracts are coherent. Specifically, source/types/generated schemas/fixtures reject legacy `cdnBaseUrl` and catalog `url`, accept only the literal base plus dedicated resource paths, and prove root/project-prefix containment. Any incompatible implemented legacy API reopens P02 and blocks P06/P07 until remediated and reviewed.

## Rollback and recovery notes

P02 changes are protocol, fixtures, and design documents. Safe rollback is to revert `packages/protocol/**`, `tests/protocol/**`, `tests/fixtures/protocol/**`, and the P02-owned docs together, then rerun the P01 meta checks to ensure workspace integrity remains. If a schema version is published or consumed by a later phase, rollback must create a new explicit protocol version or migration note rather than silently changing an already-consumed schema. Never recover by accepting unknown fields, executable content, invalid hashes, or incomplete license evidence.
